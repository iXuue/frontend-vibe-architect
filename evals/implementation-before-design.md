# Eval: Implementation Before Design

## User Input

Just start coding the frontend. Make it modern and futuristic. You decide the rest.

## Expected Routing

- Read `flows/intake.md`.
- Produce `output/requirement-summary.md`.
- Read `rules/examples-are-not-requirements.md` if the visual language is vague.
- Stop before implementation because product goal, target platform, screens, and workflows are missing.

## Expected Track

- No final track should be chosen.
- Candidate tracks may be listed only if framed as assumptions.

## Expected Strategy

- No final strategy should be chosen.
- Candidate strategies may be mentioned only as examples after the agent explains that product goal, platform, screens, workflows, and success criteria are missing.
- Do not choose `strategy/expression-first.md` or `strategy/immersive-first.md` from "modern and futuristic" alone.

## Expected Modes

- No final mode should be chosen.
- `modes/dynamic-future.md` may be mentioned only as a possible visual direction after requirements exist.

## Expected Rules

- `rules/examples-are-not-requirements.md`.
- `rules/requirement-change-control.md` if this request is part of modifying the skill itself.
- `rules/verification.md` if any implementation claims are made.

## Expected Output Templates

- `output/requirement-summary.md`.
- Do not use `output/frontend-execution-plan.md` until design direction and scope are clear.

## Must Ask Before Proceeding

- What product is being built.
- Target platform.
- Required screens or workflows.
- Whether the user wants design options first or has a confirmed direction.

## Must Not Do

- Do not start coding.
- Do not invent the whole product silently.
- Do not treat "modern and futuristic" as enough direction.
- Do not produce an execution plan without confirmed scope.

## Passing Output Should Include

- A clear stop reason.
- A requirement summary showing missing material information.
- One concise clarification question or a small set of required inputs.
