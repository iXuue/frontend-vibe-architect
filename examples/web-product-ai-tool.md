# Example: Web Product AI Tool

## User Request

I want to build an AI writing tool. It needs a homepage and a workspace. It should feel premium and future-facing, but it must still be easy to use.

## Expected Routing

Read:

- `flows/intake.md`
- `output/requirement-summary.md`
- `flows/choose-track.md`
- `tracks/web-product.md`
- `tracks/landing-brand.md`
- `rules/examples-are-not-requirements.md`
- `modes/dynamic-future.md`
- `modes/simple-designed.md`
- `flows/design-options.md`
- `output/design-options.md`

Primary track:

- `tracks/web-product.md`

Secondary track:

- `tracks/landing-brand.md`

Possible modes:

- `modes/simple-designed.md` for workspace clarity.
- `modes/dynamic-future.md` for homepage or brand moment.

## Expected Requirement Interpretation

- Hard requirements:
  - AI writing tool.
  - Homepage.
  - Workspace.
  - Must be easy to use.
- Visual intent:
  - Premium.
  - Future-facing.
- Reference examples:
  - None unless the user gives one later.
- Forbidden directions:
  - Do not sacrifice usability for atmosphere.
  - Do not make the whole product a cinematic landing page.
- Assumptions:
  - Homepage introduces the product.
  - Workspace is where repeated writing work happens.
- Open questions:
  - What are the core workspace actions: prompt, outline, draft, rewrite, history, export?
  - Is the homepage part of the same app or a separate marketing site?

## Expected Design Direction Shape

Generate 2-4 options such as:

- Balanced Product + Brand: quiet workspace, expressive homepage.
- Workspace First: simple designed app shell, minimal brand surface.
- Dynamic Launch + Practical Tool: stronger homepage atmosphere, controlled product UI.

Each option should explain:

- Which track dominates.
- Which mode applies to homepage and workspace.
- How the design protects repeated-use efficiency.
- Where motion is allowed and where it is restrained.

## Expected Output Shape

First output:

- Requirement summary.
- Track choice.
- Mode candidates.
- Design options.
- Recommendation and tradeoff.

Do not output implementation steps until the direction is confirmed or a low-risk recommendation is justified.

## Notes On What Not To Do

- Do not treat "future-facing" as a fixed visual style.
- Do not default to dark tech UI.
- Do not design only a landing page and ignore the workspace.
- Do not start coding before clarifying the core workspace workflow.
- Do not use Dynamic Future Mode across dense writing controls without safeguards.
