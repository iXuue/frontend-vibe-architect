# Output: Requirement Summary

## Use When

Use after `flows/intake.md` and before track selection.

## Required Shape

```markdown
## Requirement Summary

- Product goal:
- Target users:
- Target platform:
- Required pages or screens:
- Core workflows:
- Hard requirements:
- Reference examples:
- Visual intent:
- Forbidden directions:
- Technical constraints:
- Accessibility constraints:
- Acceptance criteria:
- Assumptions:
- Open questions:
- Potential conflicts:
```

## Do Not Omit

- Do not omit `Reference examples` when the user used example-based language.
- Do not merge assumptions into hard requirements.
- Do not hide conflicts.
- Do not proceed to design options if a material open question would change scope, platform, or acceptance criteria.
