# Rule: Reference Adaptation

## Use When

Use whenever the skill learns from installed skills, external websites, design systems, UI examples, animation libraries, notes, screenshots, or product references.

## Rule

References are learning material, not copy sources.

The skill must transform references into this user's own workflow, vocabulary, decision rules, output fields, and verification checks.

## If

If a reference contains a useful idea, first identify what type of idea it is:

- Product principle.
- Layout principle.
- Interaction principle.
- Motion principle.
- Visual variable.
- Accessibility constraint.
- Verification rule.
- Anti-pattern.
- Output format clue.

If the reference is only visually attractive but does not improve product fit, usability, implementation quality, or verification, do not carry it forward.

## Then

Convert the useful idea into one of:

- A decision rule.
- A checklist item.
- A design option criterion.
- A track priority.
- A mode safeguard.
- A verification check.
- An output field.

Combine several references into one coherent system. Do not let one reference dominate the skill's structure.

## Do

- Extract principles, not prose.
- Rewrite in the vocabulary of this skill.
- Keep the user's goal as the filter: read requirements, classify the frontend surface, generate options, and produce execution-ready steps.
- State why a borrowed principle helps the product.
- Prefer general rules that can work across many frontend scenarios.
- Add original rules where references do not cover this user's workflow.

## Do Not

- Do not copy distinctive wording.
- Do not copy complete section structures.
- Do not copy examples as-is.
- Do not copy prompts or templates as-is.
- Do not preserve another skill's command model unless it directly fits this skill.
- Do not add references that make the skill more complicated without improving decisions.

## Stop And Ask

Stop and ask the user when:

- A reference suggests a direction that conflicts with confirmed requirements.
- A reference would introduce a new dependency, workflow, or platform promise.
- The user asks to copy a proprietary design, brand, or product interface too closely.
- It is unclear whether the user wants inspiration, adaptation, or close matching.

## Output Impact

When reference material affects the result, outputs should state the adapted principle, not the copied source.

Use wording like:

- "This direction borrows the principle of..."
- "The useful variable here is..."
- "I am translating that reference into..."

Avoid wording that implies direct replication.
